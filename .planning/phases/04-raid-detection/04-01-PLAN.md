Create `src/guardian/raid_detection.py` implementing join tracking with sliding time window and account age analysis:

**JoinTracker class:**
- `__init__(window_seconds=30)`: Initialize with configurable window
- `guild_joins`: dict[guild_id] -> deque of (timestamp, member) tuples
- `add_join(guild_id, member)`: Add join to window, auto-cleanup expired entries (CRITICAL: use deque.popleft() to prevent memory leak)
- `get_recent_joins(guild_id)`: Return list of members in window, clean expired first
- `get_join_count(guild_id)`: Return count of joins in window
- Private `_cleanup_expired(guild_id)`: Remove entries older than cutoff using popleft()

**analyze_account_age_distribution function:**
- Takes: list[discord.Member], threshold_days (default 7)
- Returns: dict with total, new_accounts, percentage, threshold_days
- Calculate account age using: `(datetime.now(timezone.utc) - member.created_at).days`
- Use timezone.utc for all datetime comparisons (avoid timezone issues from Phase 3)

**Pattern from RESEARCH.md:**
Use collections.deque for O(1) append/popleft operations. Store (timestamp, member) tuples to enable age-based filtering. Clean expired entries on every add_join AND get_recent_joins call to prevent memory leak.

**DO NOT:**
- Use list instead of deque (O(n) vs O(1) for window cleanup)
- Store members without timestamps (can't expire old entries)
- Forget timezone.utc (causes account age calculation bugs)
- Skip cleanup logic (causes memory leak)

  <name>Task 2: Create lockdown manager with slowmode control</name>
  <files>src/guardian/raid_lockdown.py</files>
  <action>
Create `src/guardian/raid_lockdown.py` implementing raid lockdown state management:

**RaidLockdownManager class:**
- `__init__(recovery_seconds=900)`: 15 minutes auto-recovery
- `lockdown_state`: dict[guild_id] -> {"active": bool, "activated_at": datetime, "task": asyncio.Task}
- `async activate_lockdown(guild, alert_channel)`: Enable slowmode on all channels except ["verify", "security-logs"], send alert embed, start auto-recovery task, return bool (False if already locked)
- `async deactivate_lockdown(guild)`: Disable slowmode on all channels, cancel auto-recovery task if running, clear state, send recovery alert
- Private `async _auto_recover(guild_id)`: Sleep for recovery_seconds, then call deactivate_lockdown

**Slowmode functions (module-level):**
- `async enable_slowmode(channel, delay_seconds)`: Call `channel.edit(slowmode_delay=delay_seconds, reason="Raid lockdown activated")`, catch discord.Forbidden and discord.HTTPException, return bool
- `async disable_slowmode(channel)`: Call `channel.edit(slowmode_delay=0, reason="Raid lockdown deactivated")`, catch errors, return bool

**Lockdown activation logic:**
1. Check if already locked (return False)
2. Enable slowmode=5 on all text_channels except verify/security-logs
3. Send red embed alert to #security-logs: "🚨 Raid Detected - Lockdown Activated"
4. Create asyncio.create_task for _auto_recover
5. Store task reference in lockdown_state for cancellation

**Deactivation logic:**
1. Cancel task if exists: `task.cancel()`
2. Disable slowmode on all channels
3. Delete lockdown_state entry
4. Send green embed: "Lockdown Deactivated"

**Pattern from RESEARCH.md:**
Always store task reference to allow manual cancellation. Use `channel.edit(slowmode_delay=...)` not direct property assignment. Max slowmode is 21600 seconds, use 5s for lockdown.

**DO NOT:**
- Forget to cancel auto-recovery task on manual deactivation (causes re-lockdown bug)
- Apply slowmode to verify/security-logs channels (mods need to respond)
- Set slowmode too high (5 seconds is sufficient, 21600 makes server unusable)
